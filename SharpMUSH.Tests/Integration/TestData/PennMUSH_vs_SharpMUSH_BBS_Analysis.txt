==============================================================================
PENNMUSH vs SHARPMUSH BBS COMPARISON ANALYSIS
==============================================================================

==============================================================================
1. DEBUG OUTPUT FORMAT
==============================================================================

PennMUSH DEBUG format (from actual output):
  #4! hasflag(%#,wizard) :
  #4!  %# => #1
  #4! hasflag(%#,wizard) => 1

  - Uses spaces for nesting indentation (1 space per depth level)
  - Shows function call with ' :' suffix before evaluation  
  - Shows substitution result with ' => ' separator
  - Shows register expansion inline: %# => #1
  - Does NOT show [Registers: ...] or [Iter-Registers: ...] blocks

SharpMUSH DEBUG format (from test output):
  #4! hasflag(%#,wizard) :
  #4! [Registers: %0:1/1, %1:1/1]
  #4! hasflag(%#,wizard) => 1
  #4! [Registers: %0:1/1, %1:1/1]

  MISMATCHES:
  a) SharpMUSH emits [Registers: ...] lines after EVERY debug line
     PennMUSH does NOT emit register state between debug lines
  b) SharpMUSH emits [Iter-Registers: ...] lines during iterations
     PennMUSH does NOT emit iteration register state
  c) Both use same '#N! ' prefix format - MATCH
  d) Both use ' :' for pre-eval and ' => ' for result - MATCH

==============================================================================
2. VERBOSE OUTPUT FORMAT
==============================================================================

PennMUSH VERBOSE format:
  #1] @set me=quiet
  #4] @switch hasflag(%#,wizard)=1, {...}
  #4] @create %0
  #4] @wait 1={...}

  - Uses '#N] ' prefix
  - Shows the FULL command text BEFORE evaluation
  - For @switch, shows the entire command including branches

SharpMUSH VERBOSE format:
  #1] @set me=quiet
  #4] @switch hasflag(%#,wizard)=1, {...}

  MATCH: Same format '#N] command'

==============================================================================
3. PUPPET FLAG
==============================================================================

PennMUSH:
  @set #1=PUPPET => "PUPPET - I don't recognize that flag."
  (PUPPET is only for Things, not Players)

  @set #3=PUPPET (bbpocket, a Thing):
  "bbpocket is now listening."
  "bbpocket> bbpocket is now listening."
  "bbpocket> bbpo..."

  PUPPET on a thing causes the object's owner to see what the object
  "hears" prefixed with "objectname> ". This produces PUPPET relay messages.

SharpMUSH:
  @set #1=PUPPET => (needs verification - may silently accept or error)
  @set #3=PUPPET => Sets the PUPPET flag

  MISMATCHES:
  a) PennMUSH rejects PUPPET on Players; SharpMUSH behavior needs verification
  b) PennMUSH PUPPET relay: "objectname> message" format
     SharpMUSH PUPPET relay needs comparison

==============================================================================
4. ATTRIBUTE SET NOTIFICATIONS FORMAT
==============================================================================

PennMUSH (when VERBOSE object sets attributes):
  BBS - Myrddin's Global BBS v4.0.6> bbpocket/GROUPS - Set.
  BBS - Myrddin's Global BBS v4.0.6> TestGroup/OWN - Set.
  BBS - Myrddin's Global BBS v4.0.6> TestGroup - SAFE set.
  BBS - Myrddin's Global BBS v4.0.6> TestGroup/LAST_MOD - Set.
  BBS - Myrddin's Global BBS v4.0.6> TestGroup/CANREAD - Set.

  Format: "objectname> targetobject/ATTRNAME - Set."
  For flags: "objectname> targetobject - FLAGNAME set."

SharpMUSH:
  "I can't see that here" (×5 during +bbnewgroup)
  "I can't see that here" (×7 during +bbpost)
  
  CRITICAL MISMATCH: SharpMUSH fails to set attributes on objects
  using & command during @wait callback evaluation. PennMUSH
  successfully sets them with proper notification format.
  
  The notification format for successful attribute sets uses the
  PUPPET relay prefix: "objectname> target/ATTR - Set."

==============================================================================
5. strmatch(*=*) PARSING
==============================================================================

PennMUSH:
  #4! strmatch(%1,*=*) :
  #4!  %1 => Title Goes Here=Body of the test post.
  #4! strmatch(%1,*=*) => 1

  The *=* pattern is correctly passed to strmatch() as a literal
  glob pattern. The equals sign is NOT treated as a delimiter.

SharpMUSH:
  #4! strmatch(%1,*<missing CPAREN><missing CPAREN> :
  #4! strmatch(%1,*<missing CPAREN><missing CPAREN> => 0

  CRITICAL MISMATCH: The '=' inside strmatch(%1,*=*) is being
  parsed as an argument separator/equals delimiter by the parser.
  This causes the closing parenthesis to be consumed as part of
  the equals-separated value, resulting in <missing CPAREN> errors.
  The strmatch call effectively becomes strmatch(%1,*) with the
  =*) being lost, so it returns 0 instead of 1.

==============================================================================
6. fn_makelist / fn_maplist ATTRIBUTE LOOKUP
==============================================================================

PennMUSH:
  #3!    map(me/fn_maplist,secure(%0)) :
  #3!     secure(%0) => 1
  #3!     switch(strmatch(%0,?*-*),1,lnum(before(%0,-),after(%0,-)),%0) :
  #3!      strmatch(%0,?*-*) => 0
  #3!      %0 => 1
  #3!     switch(strmatch(%0,?*-*),1,...) => 1
  #3!    map(me/fn_maplist,secure(%0)) => 1

  PennMUSH successfully resolves map(me/fn_maplist,...) where 'me'
  resolves to bbpocket (#3), and FN_MAPLIST attribute is found.

SharpMUSH:
  #3!  map(me/fn_maplist,secure(%0)) => #-1 NO SUCH ATTRIBUTE

  CRITICAL MISMATCH: In the context of u(#3/fn_makelist,...),
  the executor is set to #3 (bbpocket), so 'me' should resolve
  to #3. But map(me/fn_maplist,...) fails to find the FN_MAPLIST
  attribute on bbpocket.
  
  Root cause: Either 'me' doesn't resolve correctly in the map()
  context, or the attribute name lookup is case-sensitive when it
  shouldn't be, or the attribute was not properly stored during
  installation.

==============================================================================
7. fn_b36-to-b10 / fn_inc_next_mess BASE-36 FUNCTIONS
==============================================================================

PennMUSH:
  #3!     u(fn_b36-to-b10,%0) :
  #3!      %0 => 1
  #3!      [setq(0,ucstr(%0))][setq(1,strlen(%q0))][fold(me/addem,...)] :
  #3!       ... (complex base-36 calculation)
  #3!     u(fn_b36-to-b10,%0) => 1
  #3!    add(u(fn_b36-to-b10,%0),1) => 2
  #3!   u(fn_b10-to-b36,add(u(fn_b36-to-b10,%0),1)) => 2
  #4!  ulocal(#3/fn_inc_next_mess,%q1) => 2

  PennMUSH correctly evaluates the base-36 conversion functions.
  fn_b36-to-b10 converts '1' to 1, then adds 1 to get 2,
  then fn_b10-to-b36 converts 2 back to '2'.

SharpMUSH:
  #3!  u(fn_b36-to-b10,%0) => #-1 INVALID OBJECT/ATTRIBUTE VALUE
  #3!  add(u(fn_b36-to-b10,%0),1) => #-1 ARGUMENTS MUST BE NUMBERS
  #3!  u(fn_b10-to-b36,add(u(fn_b36-to-b10,%0),1)) => #-1 INVALID OBJECT/ATTRIBUTE VALUE
  #4! ulocal(#3/fn_inc_next_mess,%q1) => #-1 INVALID OBJECT/ATTRIBUTE VALUE

  CRITICAL MISMATCH: The fn_b36-to-b10 attribute function fails
  entirely. This might be related to the fn_maplist issue (#6) -
  if attribute lookup within u() doesn't properly resolve 'me'
  references or hyphenated attribute names, the entire base-36
  system fails. This cascades to fn_inc_next_mess, preventing
  the BBS from incrementing message IDs.

==============================================================================
8. hasflag(%0,DARK) ON OBJECTS
==============================================================================

PennMUSH:
  #3! hasflag(%0,DARK) :
  #3!  %0 => #1
  #3! hasflag(%0,DARK) => 0

  PennMUSH successfully checks if #1 has the DARK flag (it doesn't).

SharpMUSH:
  #3! hasflag(%0,DARK) => #-1 CAN'T SEE THAT HERE

  MISMATCH: hasflag() fails with "CAN'T SEE THAT HERE" when
  checking flags on an object. This suggests a visibility/location
  check is failing - the object executing the hasflag check can't
  "see" the target object. PennMUSH's hasflag() doesn't require
  the object to be visible/in the same location.

==============================================================================
9. PERMISSION DENIED ON &bb_read
==============================================================================

PennMUSH:
  #4] &bb_read %#=[setunion(get(%#/bb_read),%q1)]
  BBS - Myrddin's Global BBS v4.0.6> Permission denied.

  NOTE: PennMUSH also gets "Permission denied" for &bb_read when
  mbboard (#4) tries to set an attribute on #1 (the player).
  This is EXPECTED behavior - a thing (mbboard) cannot set
  attributes on a player unless it has special powers.

SharpMUSH:
  #-1 NO PERMISSION TO SET ATTRIBUTE

  MINOR MISMATCH: The error message text differs:
  PennMUSH: "Permission denied."
  SharpMUSH: "#-1 NO PERMISSION TO SET ATTRIBUTE"
  
  Both correctly deny the permission, but the error format differs.

==============================================================================
10. @DOLIST ITERATION REGISTER (%iL / ##)
==============================================================================

PennMUSH:
  @dolist uses ## as the iteration variable in the body.
  When the @dolist list evaluates to "1", the body runs once
  with ## = 1, and the iteration works correctly.

  In verbose/debug mode, PennMUSH shows the @dolist command
  AFTER evaluating the list expression, then shows each
  iteration step. The ## substitution in commands like:
    @switch [and(lte(1,words(get(%q0/MESS_LST))),isnum(1),gt(1,0))]=0,...
  shows the numeric value 1 directly substituted.

SharpMUSH:
  [Iter-Registers: %iL:#-1]
  
  The iteration register %iL (equivalent to ##) contains #-1
  instead of the actual list element. This causes all numeric
  checks (lte, isnum, gt) to fail:
    lte(%iL,words(...)) => #-1 ARGUMENTS MUST BE NUMBERS
    gt(%iL,0) => #-1 ARGUMENTS MUST BE NUMBERS
  
  Root cause: The fn_makelist function returns #-1 (because of
  the fn_maplist attribute lookup failure), and this #-1 value
  becomes the iteration variable.
  
  CASCADE: This causes "Message 1/#-1 (TestGroup/#-1) does not exist."


==============================================================================
SUMMARY OF MISMATCHES
==============================================================================

CRITICAL (blocks BBS functionality):
1. [Registers:] blocks emitted after every debug line - NOT in PennMUSH
2. strmatch(*=*) parsing - '=' treated as delimiter, not literal
3. fn_maplist/fn_makelist attribute lookup - 'me' resolution or attr name issue  
4. fn_b36-to-b10 attribute function evaluation fails entirely
5. hasflag(%0,DARK) returns CAN'T SEE THAT HERE instead of 0/1
6. "I can't see that here" during & attribute set operations
7. %iL iteration register gets #-1 from failed fn_makelist

MINOR (formatting differences):
8. Permission denied message format differs
9. PUPPET flag acceptance on Players differs (PennMUSH rejects, SharpMUSH may accept)
10. BBS at "0%" vs "0.0%" capacity (PennMUSH shows 0.0%, SharpMUSH shows 0%)
