## Error Pattern Summary

### Pattern A: Empty Command Arguments (7 failures)
Tests: Flag_List, Power_List, SuggestListCommand, Entrances, Search, DoBreak, BasicLambda

**Common Error**:
```
line 1:0 no viable alternative at input '<EOF>'
System.InvalidOperationException: Parser error in rule 'evaluationString'
```

**Why it fails in strict mode**:
The evaluationString grammar rule has two alternatives but no empty option:
```antlr
evaluationString:
      function explicitEvaluationString?
    | explicitEvaluationString
;
```

When input is empty (EOF), AdaptivePredict cannot determine which alternative to choose because:
- First alternative: `function explicitEvaluationString?` - could match if function has no args
- Second alternative: `explicitEvaluationString` - would fail but parser doesn't know without trying

In strict mode, this prediction uncertainty causes NoViableAltException.

### Pattern B: EqSplit Empty Values (2 failures)
Tests: Test_Respond_Type_Empty, Test_Respond_Header_EmptyName

**Common Error**:
```
Parser error in startEqSplitCommand
Input like '=value' or 'key=' cannot be predicted
```

**Why it fails in strict mode**:
```antlr
startEqSplitCommand:
    (singleCommandArg (EQUALS singleCommandArg))? EOF
;
```

The optional pattern where both args can be empty creates ambiguity:
- Is `=value` an empty first arg?
- Or is it not matching the pattern at all?

### Pattern C: Leading Delimiter (1 failure)
Test: Test(]think [add(1,2)]3)

**Error**:
```
Parser cannot determine if ] is delimiter or text
```

**Why it fails in strict mode**:
The `]` character needs context predicate (like CPAREN has) to know when it's plain text vs delimiter.

## Grammar Analysis

### evaluationString Rule - Core Issue

**Current Definition**:
```antlr
evaluationString:
      function explicitEvaluationString?
    | explicitEvaluationString
;
```

**Problem**: No empty alternative, but callers assume it can handle empty input.

**Impact**: 7 out of 10 failures (70%)

**Solutions**:

**Option 1: Add Empty Alternative**
```antlr
evaluationString:
      function explicitEvaluationString?
    | explicitEvaluationString
    | /* empty - allow zero-length expressions */
;
```
Pros: Simple, fixes all 7 tests
Cons: Could hide real errors, needs visitor validation

**Option 2: Lookahead Predicates**
```antlr
evaluationString:
      function explicitEvaluationString?
    | explicitEvaluationString
    | {InputStream.LA(1) == COMMAWS || InputStream.LA(1) == CPAREN || 
       InputStream.LA(1) == SEMICOLON || InputStream.LA(1) == CBRACE || 
       InputStream.LA(1) == Eof}? /* empty when followed by delimiter */
;
```
Pros: Explicit, predictable, safe
Cons: More complex, needs maintenance

**Option 3: Use ? Operator in Callers**
Change calling rules to use `evaluationString?` instead of `evaluationString`
Pros: Follows ANTLR4 best practices
Cons: Requires many grammar changes, visitor updates

### startEqSplitCommand Rule - Secondary Issue

**Current Definition**:
```antlr
startEqSplitCommand:
    {lookingForCommandArgEquals = true;} (singleCommandArg (
        EQUALS {lookingForCommandArgEquals = false;} singleCommandArg
    ))? EOF
;
```

**Problem**: Optional pattern with optional inner components creates prediction ambiguity.

**Impact**: 2 out of 10 failures (20%)

**Solution**: Split into separate rules for different input patterns:
```antlr
startEqSplitCommand:
      singleCommandArg EQUALS singleCommandArg EOF   // key=value
    | EQUALS singleCommandArg EOF                     // =value
    | singleCommandArg EQUALS EOF                     // key=
    | singleCommandArg EOF                            // key (no equals)
    | EOF                                             // empty
;
```

### Leading Delimiter - Minor Issue

**Problem**: `]` character at string start not handled like `)` and other delimiters.

**Impact**: 1 out of 10 failures (10%)

**Solution**: Add to beginGenericText with predicate:
```antlr
beginGenericText:
      { inFunction == 0 }? CPAREN
    | { inBraceDepth == 0 }? CBRACK  // Add this
    | ...
;
```

## Production Impact Assessment

**Q: Do these failures affect production use?**  
**A: No.** All 10 tests pass with normal error recovery strategy.

**Q: Why do they pass in normal mode?**  
**A: Normal mode** uses error recovery:
1. Parser tries first alternative
2. If it fails, backtracks
3. Tries second alternative
4. If both fail, inserts/deletes tokens to recover
5. Continues parsing

**Strict mode** fails fast at prediction phase - doesn't try alternatives or recover.

**Q: Should we fix these ambiguities?**  
**A: Optional.** Benefits:
- Faster parsing (no backtracking)
- Clearer grammar semantics
- Better error messages
- Future-proofing

But current grammar works correctly in production.
